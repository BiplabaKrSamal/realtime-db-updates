Why WAL over LISTEN/NOTIFY?

CriterionLISTEN / NOTIFYWAL Logical Replication ✓ Chosen
Payload size8 KB hard cap — silent truncation riskUnlimited — full row, any column type
Old row on UPDATENot available — triggers see only new rowFull OLD row always available for diffing
Crash recoveryEvents lost while server is downLSN acknowledgement — replay from last position
Schema changesTrigger required on every tableZero schema changes — publication covers any table
Bulk load / restorepg_restore suppresses triggers — silent gapAll WAL changes captured regardless of source
DB overheadPer-row trigger executes function on each writeReplication is a first-class PG feature, minimal overhead
